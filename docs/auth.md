# SmartDB 認証（APIキー）設計メモ

このメモは実装の指針であり、コードそのものは含めない。
最初の段階では認証のみを実装し、権限の出し分けは次の段階に回す。

## スコープ

第一段階では、リクエストが有効なトークンを持つかどうかだけを検証する。
検証を通れば、そのキーで許される範囲の操作はすべて許可する。

権限の3段階（Admin、Read-Write、Read-Only）と、SQLの種別（`QueryJudge` の結果）を突き合わせた拒否判定は、第二段階に回す。
ただしスキーマには最初から `role` 列を持たせ、第二段階で列を追加せずに済むようにする。

## キーの単位

キーには2種類ある。

**システム管理キー**：プロジェクトに紐づかない。プロジェクトの作成と削除、および全プロジェクトへのアクセスができる。

**プロジェクト単位キー**：特定のプロジェクトに紐づく。そのプロジェクトに対する操作だけができる。

データ上は、プロジェクトに紐づくかどうかを `project_id` が NULL かどうかで表す。
NULL ならシステム管理キー、値が入っていればそのプロジェクト専用のキーとする。

## データモデル

APIキーはシステムDB（`system.db`）の `api_keys` テーブルで管理する。

```sql
CREATE TABLE IF NOT EXISTS api_keys (
    id           TEXT PRIMARY KEY,
    project_id   TEXT REFERENCES projects(id),
    name         TEXT NOT NULL,
    token_hash   TEXT NOT NULL UNIQUE,
    role         TEXT NOT NULL CHECK (role IN ('admin', 'read_write', 'read_only')),
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    revoked_at   DATETIME
);
```

各列の意味は次のとおり。

**id**：キーの識別子。一覧や失効で参照する。
**project_id**：紐づくプロジェクト。NULL ならシステム管理キー。
**name**：人が見分けるためのラベル。
**token_hash**：トークンのハッシュ。平文は保存しない。
**role**：権限レベル。第一段階では保存するだけで、判定には使わない。
**revoked_at**：失効した時刻。NULL なら有効。

失効は行の削除ではなく `revoked_at` に時刻を入れる方式とする。
プロジェクトの状態管理と同じく、履歴を残しつつ無効化できる。

## トークンのライフサイクル

トークンは次の流れで扱う。

発行時は、`crypto/rand` で十分な長さ（32バイト程度）の乱数を生成し、文字列に変換する。
判別しやすいよう `sdb_` のような接頭辞を付けてよい。
この平文トークンは、発行のレスポンスで一度だけ返す。

保存時は、平文をそのまま保存せず、SHA-256 のハッシュを取って `token_hash` に入れる。
トークンは高エントロピーの乱数なので、パスワード用の遅いハッシュ（bcrypt など）は不要で、SHA-256 で足りる。

検証時は、リクエストの `Authorization: Bearer <token>` からトークンを取り出し、同じ手順でハッシュ化する。
そのハッシュで `api_keys` を引き、`revoked_at IS NULL` の行が見つかれば有効とする。
見つかった行から `project_id` と `role` が得られる。

失効時は、対象の行の `revoked_at` に現在時刻を入れる。

平文トークンは、発行時のレスポンス以外のどこにも出さない。
ログにも一覧APIのレスポンスにも含めない。

## ブートストラップ

最初のシステム管理キーをどう発行するかが問題になる。
プロジェクトの作成にはシステム管理キーが要るが、初回起動時にはまだ1本も存在しない。

起動時に、システム管理キー（`project_id IS NULL`）が1本も無ければ、その場で1本生成する。
生成した平文トークンを標準出力かログに一度だけ出し、運用者が控える。
2回目以降の起動では、すでに存在するので何もしない。

## 認証ミドルウェア

検証は各ハンドラの手前のミドルウェアで行う。
すでにある `internal/handler` のミドルウェアと同じ層に置く。

ミドルウェアの責務は次のとおり。

`Authorization` ヘッダから Bearer トークンを取り出す。
トークンをハッシュ化して `api_keys` を引き、有効なキーかを判定する。
有効なら、得られた `project_id` と `role` をリクエストの `context` に載せ、次のハンドラへ渡す。
無効なら 401 を返し、次へは進めない。

エンドポイントによって要求するキーの種類を変える。

プロジェクト管理（`POST /projects`、`DELETE /projects/{project}`、一覧）は、システム管理キーを要求する。

プロジェクト配下（`/projects/{project}/...`）は、システム管理キー、またはそのプロジェクトに紐づくプロジェクト単位キーを要求する。
プロジェクト単位キーの場合は、キーの `project_id` がパスの `{project}` と一致することを確認する。

## APIキー管理エンドポイント

仕様書に沿って3つを用意する。

発行：`POST /api/v1/projects/{project}/apikeys`
一覧：`GET /api/v1/projects/{project}/apikeys`
失効：`DELETE /api/v1/projects/{project}/apikeys/{id}`

一覧のレスポンスには `token_hash` を含めない。
返すのは `id`、`name`、`role`、`created_at`、`revoked_at` など、秘密でない情報だけにする。

## 実装チェックリスト（第一段階）

1. システムDBの初期化に `api_keys` のDDLを追加する。
2. トークン生成（乱数→文字列）とハッシュ化（SHA-256）の処理を用意する。
3. キー発行の処理を書く。生成し、ハッシュを保存し、平文を一度だけ返す。
4. 起動時ブートストラップを書く。システム管理キーが無ければ生成してログに出す。
5. 認証ミドルウェアを書く。ヘッダ解析、照合、`context` への付与、401応答。
6. ルータで各エンドポイントにミドルウェアを被せ、要求するキーの種類を出し分ける。
7. APIキー管理エンドポイント（発行、一覧、失効）を追加する。
